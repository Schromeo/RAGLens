# Trace Data Model

## Purpose

This document defines the initial trace/span data model for RAGLens v0.1.

It is the shared contract between:

- Python SDK
- Go Collector
- SQLite storage
- React Dashboard

RAGLens exposes a simple developer-facing API, but internally stores RAG pipeline activity as traces and spans.

This keeps the MVP easy to use while preserving a clean path toward future LLM/Agent observability features.

---

## Design Principles

### 1. Local-first

RAGLens v0.1 is designed to run locally.

The default storage backend is SQLite.

No cloud service, authentication, team workspace, or external infrastructure is required.

### 2. Simple user API, structured internal model

The Python SDK should feel simple:

```python
from raglens import trace

with trace("refund-policy-qa") as t:
    t.retrieval(query="What is the refund window?", chunks=[...])
    t.llm(model="gpt-4o-mini", prompt="...", response="...")
````

Internally, RAGLens stores this as:

```text
Trace
  ├── Retrieval Span
  └── LLM Span
```

### 3. RAG-first, AgentOps-ready

RAGLens v0.1 focuses on RAG debugging.

However, the model should support future span types such as:

* tool_call
* agent_step
* memory_read
* rerank
* eval
* guardrail
* embedding

### 4. JSON-first flexibility

Most input, output, and metadata fields are stored as JSON.

This keeps the schema flexible while the product is still evolving.

Common fields can be normalized later if needed.

### 5. Easy to inspect

The model should be easy to inspect in:

* SQLite
* API responses
* React Dashboard
* raw JSON exports

## Core Concepts

## Trace

A trace represents one complete RAG request.

Example:

```text
User asks:
"What is the refund policy?"

The application:
1. Retrieves relevant chunks
2. Builds a prompt
3. Calls an LLM
4. Returns an answer
5. RAGLens generates warnings
```

That full request is one trace.

### Trace responsibilities

A trace stores:

* The user query
* Final answer
* Overall status
* Timing information
* App metadata
* Child spans
* Generated warnings

---

## Span

A span represents one step inside a trace.

In v0.1, the main span types are:

* retrieval
* prompt
* llm
* custom

Example:

```text
Trace: refund-policy-qa
  ├── Span: retrieval / search_docs
  └── Span: llm / generate_answer
```

### Span responsibilities

A span stores:

* Span type
* Span name
* Input
* Output
* Metadata
* Timing information
* Error information

---

## Retrieval Chunk

A retrieval chunk represents one piece of context returned by a retriever.

Chunks usually belong to a retrieval span.

A chunk should include:

* Chunk ID
* Text
* Score
* Rank
* Source
* Document ID
* Metadata

Example:

```json
{
  "id": "chunk_1",
  "text": "Customers may request a refund within 30 days.",
  "score": 0.91,
  "rank": 1,
  "source": "refund_policy_new.md",
  "document_id": "refund_policy_new",
  "metadata": {
    "version": "2026"
  }
}
```

---

## Warning

A warning is a diagnostic result generated by RAGLens.

Warnings are derived from traces and spans.

Examples:

* No chunks were retrieved
* Top retrieval score is low
* Retrieved chunks appear to conflict
* Answer may not be grounded in the retrieved chunks
* Duplicate chunks were retrieved
* LLM call failed

Warnings are not meant to be perfect in v0.1.

They are lightweight debugging signals.

# Trace Schema

## Logical schema

```json
{
  "trace_id": "trace_01HZ...",
  "name": "refund-policy-qa",
  "status": "ok",
  "input": {
    "query": "What is the refund window?"
  },
  "output": {
    "answer": "Customers may request a refund within 14 days."
  },
  "metadata": {
    "app": "refund-policy-demo",
    "environment": "local",
    "sdk_language": "python",
    "sdk_version": "0.1.0"
  },
  "started_at": "2026-05-14T10:00:00Z",
  "ended_at": "2026-05-14T10:00:02Z",
  "duration_ms": 2000,
  "spans": [],
  "warnings": []
}
```

## Fields

| Field         | Type    | Required | Description                                      |
| ------------- | ------- | -------: | ------------------------------------------------ |
| `trace_id`    | string  |      yes | Unique trace ID                                  |
| `name`        | string  |      yes | Human-readable trace name                        |
| `status`      | string  |      yes | `ok` or `error`                                  |
| `input`       | object  |       no | Trace-level input, usually user query            |
| `output`      | object  |       no | Trace-level output, usually final answer         |
| `metadata`    | object  |       no | App, environment, SDK, and user-defined metadata |
| `started_at`  | string  |      yes | ISO-8601 timestamp                               |
| `ended_at`    | string  |       no | ISO-8601 timestamp                               |
| `duration_ms` | integer |       no | Total trace duration                             |
| `spans`       | array   |       no | Child spans                                      |
| `warnings`    | array   |       no | Generated warnings                               |

## Trace status

Allowed values for v0.1:

```text
ok
error
```

Future values may include:

```text
pending
cancelled
timeout
```

# Span Schema

## Logical schema

```json
{
  "span_id": "span_01HZ...",
  "trace_id": "trace_01HZ...",
  "parent_span_id": null,
  "type": "retrieval",
  "name": "search_docs",
  "status": "ok",
  "input": {},
  "output": {},
  "metadata": {},
  "started_at": "2026-05-14T10:00:00Z",
  "ended_at": "2026-05-14T10:00:01Z",
  "duration_ms": 1000,
  "error": null
}
```

## Fields

| Field            | Type        | Required | Description                     |
| ---------------- | ----------- | -------: | ------------------------------- |
| `span_id`        | string      |      yes | Unique span ID                  |
| `trace_id`       | string      |      yes | Parent trace ID                 |
| `parent_span_id` | string/null |       no | Parent span ID for nested spans |
| `type`           | string      |      yes | Span type                       |
| `name`           | string      |      yes | Human-readable span name        |
| `status`         | string      |      yes | `ok` or `error`                 |
| `input`          | object      |       no | Span input                      |
| `output`         | object      |       no | Span output                     |
| `metadata`       | object      |       no | Span metadata                   |
| `started_at`     | string      |      yes | ISO-8601 timestamp              |
| `ended_at`       | string      |       no | ISO-8601 timestamp              |
| `duration_ms`    | integer     |       no | Span duration                   |
| `error`          | object/null |       no | Error details if failed         |

## Span types for v0.1

```text
retrieval
prompt
llm
custom
```

## Future span types


# Retrieval Span

A retrieval span records the result of searching a knowledge source.

## Example

```json
{
  "span_id": "span_retrieval_001",
  "trace_id": "trace_001",
  "parent_span_id": null,
  "type": "retrieval",
  "name": "search_refund_docs",
  "status": "ok",
  "input": {
    "query": "What is the refund window?",
    "top_k": 3
  },
  "output": {
    "chunks": [
      {
        "id": "chunk_1",
        "text": "Customers may request a refund within 30 days.",
        "score": 0.91,
        "rank": 1,
        "source": "refund_policy_new.md",
        "document_id": "refund_policy_new",
        "metadata": {
          "version": "2026"
        }
      },
      {
        "id": "chunk_2",
        "text": "Customers may request a refund within 14 days.",
        "score": 0.84,
        "rank": 2,
        "source": "refund_policy_old.md",
        "document_id": "refund_policy_old",
        "metadata": {
          "version": "2024"
        }
      }
    ]
  },
  "metadata": {
    "retriever": "local_vector_store",
    "embedding_model": "text-embedding-3-small"
  },
  "started_at": "2026-05-14T10:00:00Z",
  "ended_at": "2026-05-14T10:00:01Z",
  "duration_ms": 1000,
  "error": null
}
```

## Retrieval span input

| Field     | Type    | Required | Description                            |
| --------- | ------- | -------: | -------------------------------------- |
| `query`   | string  |      yes | Query sent to retriever                |
| `top_k`   | integer |       no | Number of requested chunks             |
| `filters` | object  |       no | Metadata filters used during retrieval |

## Retrieval span output

| Field    | Type  | Required | Description      |
| -------- | ----- | -------: | ---------------- |
| `chunks` | array |      yes | Retrieved chunks |

## Chunk schema

| Field         | Type         | Required | Description               |
| ------------- | ------------ | -------: | ------------------------- |
| `id`          | string       |      yes | Chunk ID                  |
| `text`        | string       |      yes | Chunk text                |
| `score`       | number/null  |       no | Retrieval score           |
| `rank`        | integer/null |       no | Retrieval rank            |
| `source`      | string/null  |       no | Source file or document   |
| `document_id` | string/null  |       no | Parent document ID        |
| `metadata`    | object       |       no | Additional chunk metadata |

# Prompt Span

A prompt span records prompt construction.

This is optional in v0.1, but useful for debugging how retrieved chunks were converted into an LLM input.

## Example

```json
{
  "span_id": "span_prompt_001",
  "trace_id": "trace_001",
  "parent_span_id": null,
  "type": "prompt",
  "name": "build_rag_prompt",
  "status": "ok",
  "input": {
    "query": "What is the refund window?",
    "chunk_ids": ["chunk_1", "chunk_2"]
  },
  "output": {
    "prompt": "Answer the question using only the provided context..."
  },
  "metadata": {
    "template_name": "basic_rag_prompt"
  },
  "started_at": "2026-05-14T10:00:01Z",
  "ended_at": "2026-05-14T10:00:01Z",
  "duration_ms": 50,
  "error": null
}
```

# LLM Span

An LLM span records a model call.

## Example

```json
{
  "span_id": "span_llm_001",
  "trace_id": "trace_001",
  "parent_span_id": null,
  "type": "llm",
  "name": "generate_answer",
  "status": "ok",
  "input": {
    "model": "gpt-4o-mini",
    "prompt": "Answer the user question using the retrieved context."
  },
  "output": {
    "response": "Customers may request a refund within 14 days."
  },
  "metadata": {
    "provider": "openai",
    "temperature": 0.2,
    "input_tokens": 320,
    "output_tokens": 80,
    "total_tokens": 400,
    "latency_ms": 900
  },
  "started_at": "2026-05-14T10:00:01Z",
  "ended_at": "2026-05-14T10:00:02Z",
  "duration_ms": 900,
  "error": null
}
```

## LLM span input

| Field        | Type   | Required | Description      |
| ------------ | ------ | -------: | ---------------- |
| `model`      | string |      yes | Model name       |
| `prompt`     | string |       no | Prompt text      |
| `messages`   | array  |       no | Chat messages    |
| `parameters` | object |       no | Model parameters |

At least one of `prompt` or `messages` should be present.

## LLM span output

| Field          | Type   | Required | Description                    |
| -------------- | ------ | -------: | ------------------------------ |
| `response`     | string |      yes | Model response text            |
| `raw_response` | object |       no | Optional raw provider response |

## LLM metadata

| Field           | Type    | Required | Description          |
| --------------- | ------- | -------: | -------------------- |
| `provider`      | string  |       no | LLM provider         |
| `temperature`   | number  |       no | Sampling temperature |
| `input_tokens`  | integer |       no | Input token count    |
| `output_tokens` | integer |       no | Output token count   |
| `total_tokens`  | integer |       no | Total token count    |
| `latency_ms`    | integer |       no | LLM call latency     |

# Warning Schema

## Logical schema

```json
{
  "warning_id": "warn_001",
  "trace_id": "trace_001",
  "span_id": "span_retrieval_001",
  "type": "conflicting_chunks",
  "severity": "high",
  "message": "Retrieved chunks contain conflicting refund windows: 30 days and 14 days.",
  "details": {
    "values": ["30 days", "14 days"],
    "chunk_ids": ["chunk_1", "chunk_2"]
  },
  "created_at": "2026-05-14T10:00:02Z"
}
```

## Fields

| Field        | Type        | Required | Description              |
| ------------ | ----------- | -------: | ------------------------ |
| `warning_id` | string      |      yes | Unique warning ID        |
| `trace_id`   | string      |      yes | Parent trace ID          |
| `span_id`    | string/null |       no | Related span ID          |
| `type`       | string      |      yes | Warning type             |
| `severity`   | string      |      yes | Warning severity         |
| `message`    | string      |      yes | Human-readable message   |
| `details`    | object      |       no | Machine-readable details |
| `created_at` | string      |      yes | ISO-8601 timestamp       |

## Warning types for v0.1

```text
no_retrieved_chunks
low_retrieval_score
conflicting_chunks
answer_not_grounded
duplicate_chunks
llm_error
retrieval_error
```

## Warning severity

```text
low
medium
high
```

# SQLite Schema

RAGLens v0.1 uses SQLite as the default local storage backend.

## traces table

```sql
CREATE TABLE IF NOT EXISTS traces (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    status TEXT NOT NULL,
    input_json TEXT,
    output_json TEXT,
    metadata_json TEXT,
    started_at TEXT NOT NULL,
    ended_at TEXT,
    duration_ms INTEGER,
    created_at TEXT NOT NULL
);
```

## spans table

```sql
CREATE TABLE IF NOT EXISTS spans (
    id TEXT PRIMARY KEY,
    trace_id TEXT NOT NULL,
    parent_span_id TEXT,
    type TEXT NOT NULL,
    name TEXT NOT NULL,
    status TEXT NOT NULL,
    input_json TEXT,
    output_json TEXT,
    metadata_json TEXT,
    started_at TEXT NOT NULL,
    ended_at TEXT,
    duration_ms INTEGER,
    error_json TEXT,
    created_at TEXT NOT NULL,
    FOREIGN KEY(trace_id) REFERENCES traces(id)
);
```

## warnings table

```sql
CREATE TABLE IF NOT EXISTS warnings (
    id TEXT PRIMARY KEY,
    trace_id TEXT NOT NULL,
    span_id TEXT,
    type TEXT NOT NULL,
    severity TEXT NOT NULL,
    message TEXT NOT NULL,
    details_json TEXT,
    created_at TEXT NOT NULL,
    FOREIGN KEY(trace_id) REFERENCES traces(id),
    FOREIGN KEY(span_id) REFERENCES spans(id)
);
```

## Recommended indexes

```sql
CREATE INDEX IF NOT EXISTS idx_traces_started_at
ON traces(started_at);

CREATE INDEX IF NOT EXISTS idx_spans_trace_id
ON spans(trace_id);

CREATE INDEX IF NOT EXISTS idx_warnings_trace_id
ON warnings(trace_id);

CREATE INDEX IF NOT EXISTS idx_warnings_span_id
ON warnings(span_id);
```

# Error Model

If a span fails, the span should have:

```json
{
  "status": "error",
  "error": {
    "type": "TimeoutError",
    "message": "Retriever timed out after 3000ms",
    "stack": "optional stack trace"
  }
}
```

If a trace contains one failed span, the trace status should usually be `error`.

Future versions may support partial success.

---

# ID Format

RAGLens IDs should be string IDs.

Recommended prefixes:

```text
trace_
span_
warn_
```

Example:

```text
trace_01HZABC...
span_01HZDEF...
warn_01HZGHI...
```

The exact ID generator can be decided during SDK implementation.

Acceptable options:

* UUID
* ULID
* timestamp-based random ID

For v0.1, UUID is acceptable because it is simple and widely supported.

---

# Time Format

All timestamps should use ISO-8601 strings in UTC.

Example:

```text
2026-05-14T10:00:00Z
```

Durations should be stored in milliseconds.
